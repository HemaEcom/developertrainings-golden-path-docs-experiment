CDK Infrastructure Guide

Reference implementation: omni-web-content-frontend

Overview

Section titled “Overview”

Every HEMA micro-frontend (MFE) runs on AWS using a two-stack CDK architecture:

StackSuffixDeployed byPurpose
CI Stack-ciDeveloper (one-time cdk deploy)CodePipeline V2 that builds, tests, and deploys the runtime stack
Runtime Stack-rtThe CI pipeline (on every commit to main)ECS Fargate service, ALB, VPC Origin, Gateway Registration, Monitoring

Key rule: Pipelines do not deploy pipelines. The CI stack is deployed manually; it then deploys the runtime stack automatically.

ComponentPurpose

VPC OriginCloudFront VPC Origin for private connectivity (no public ALB)

Pipeline Stack

Section titled “Pipeline Stack”

The pipeline uses CodePipeline V2 with these stages:

  1. Source — GitHub connection via CodeStar
  2. Synth — Install deps, lint, test, build Next.js, cdk synth
  3. Deploy — Deploy the runtime stack (with manual approval for prod)
  4. GarbageCollect — Clean up stale CDK assets and ECR images

Environment Configuration (SSM Parameters)

Section titled “Environment Configuration (SSM Parameters)”

All runtime configuration is stored in AWS SSM Parameter Store:

Parameter PathPurpose
/nordcloud/ntwk/prov-mainvpc-vpcidVPC ID for the service
/hema/global/gateway/account-idGateway account ID for RAM sharing
/hema/global/authSecurity credentials (Secrets Manager)
/hema/{service}/{component}/{env}/log-levelLog level per environment
/hema/{service-name}/sanity/project-idSanity project ID
/hema/{service-name}/{env}/sanity/datasetSanity dataset per environment
/hema/{service-name}/{env}/base-urlPublic base URL

Deploying Your First Stack

Section titled “Deploying Your First Stack”

After this, every push to main triggers the pipeline which:

  1. Builds the Next.js app into a Docker image
  2. Synthesizes the CDK runtime stack
  3. Deploys the runtime stack (ECS, ALB, VPC Origin, Gateway Registration)
