Butler & Feature Sandbox Deployments

Source: omni-web-content-frontend buildspec-ci.yaml | Butler docs Runtime: Node.js 22 | CDK | CodeBuild

Overview

Section titled “Overview”

Butler is HEMA’s standard solution for feature branch deployments. It automatically creates ephemeral sandbox environments for every feature branch, enabling:

  • Realistic testing in production-like conditions
  • Stakeholder review before merging
  • Parallel development without environment conflicts
  • Automatic cleanup when branches are merged/deleted

How Butler Works

Section titled “How Butler Works”
  1. Developer pushes a feature branch (e.g., feature/new-hero-block)
  2. Butler detects the branch and triggers a CodeBuild project
  3. CodeBuild executes buildspec-ci.yaml in the repo root
  4. The buildspec installs dependencies, builds the app, and runs cdk deploy
  5. A new CloudFormation stack is created with a unique name based on the branch
  6. A preview URL is generated for testing
  7. When the branch is merged or deleted, Butler automatically destroys the stack

The buildspec-ci.yaml File

Section titled “The buildspec-ci.yaml File”

This is the file Butler uses to build and deploy your feature branch. Place it in the repo root (not in src/).

AspectDetail
Node.js versionMust be 22 (matches production)
CodeArtifact authRequired for @hema/* packages from private registry
Two npm installsRoot for CDK deps, src/ for app deps
--legacy-peer-depsNeeded for HDS RC version resolution
Branch nameLowercased and used as the stack/environment name
butlerStackTagTag that Butler uses to track and clean up stacks
--require-approval=neverAutomated deploy, no manual approval

Section titled “Environment Variables Provided by Butler”

Butler injects these variables into the CodeBuild environment:

VariableExamplePurpose
BRANCHfeature/new-hero-blockFull branch name (with slashes)

BUTLER_STACK_TAGbutler-managedTag for stack lifecycle management

Automatic Cleanup

Section titled “Automatic Cleanup”

Butler automatically destroys sandbox stacks when:

  • The feature branch is merged into main
  • The feature branch is deleted

The butlerStackTag CDK context variable is used to identify which stacks Butler manages. Stacks without this tag are not touched.

Troubleshooting

Section titled “Troubleshooting”
IssueSolution
Build fails at npm ciCheck CodeArtifact auth token is valid; ensure .npmrc is not overriding registry
Stack deploy failsCheck CloudFormation events in AWS Console for the specific error
Preview URL not workingVerify gateway registration includes the branch environment; check ALB health checks
Stack not cleaned upVerify butlerStackTag is passed to CDK deploy; check Butler logs
--legacy-peer-deps neededHDS RC versions may conflict with other deps; this flag resolves peer dep warnings
